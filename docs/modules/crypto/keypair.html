Asymmetric Encryption (KEYPAIR)

How do you talk securely with someone you’ve never met? Keypairs make it possible — no shared secret required.

Why Keypairs?

In Module 1, we learned about symmetric encryption — where the same key locks and unlocks a message. It’s fast, but it only works if you’ve already agreed on a shared key.

But what if you’re starting from scratch? That’s where keypairs come in. Instead of one secret, you have two:

  • Public Key: Can be shared with anyone. Used to encrypt messages or verify your digital signature.
  • Private Key: Kept secret. Used to decrypt messages or create a digital signature.

These keys are mathematically linked. What one locks, only the other can unlock. But knowing one doesn’t reveal the other.

What Makes This Powerful?

With a keypair, you don’t need to secretly exchange passwords ahead of time. You just publish your public key.

Anyone can encrypt a message to you, but only you can read it. That’s how HTTPS secures your connection to websites. It’s how Signal secures your chats.

Where It Comes From

This idea first appeared in 1976, when Diffie and Hellman proposed public key exchange. Then in 1978, Rivest, Shamir, and Adleman (RSA) showed how to use it in practice.

Today, keypairs power nearly all secure communication — web browsers, email, cryptocurrency wallets, even software updates.

But Who Do You Trust?

If someone sends you a public key and says, “This is your bank,” how do you know it’s true?

This is called a man-in-the-middle attack. To defend against it, we use things like certificate authorities (CAs) — trusted organizations that vouch for keys. But they reintroduce centralized control.

What Can Go Wrong?

  • Using the same key on multiple platforms can leak data.
  • Bad randomness during key generation can make keys guessable.
  • Poor implementation — like skipping verification checks — opens holes.

The math is solid. But cryptography lives inside software — and that’s where things often break.
